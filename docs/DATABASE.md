# База данных и схема

## Что считать источником правды

В этом репозитории схема хранится в двух видах:

- `database/migrations/prod.sql` — основной SQL-дамп схемы.
- `database/exports/*.json` — экспортированные артефакты таблиц, индексов, функций, триггеров и enum'ов.

Для прикладного кода важнее всего таблицы из схемы `public`. `auth.*` и `realtime.*` присутствуют как часть Supabase-дампа, но обычно не редактируются в рамках логики приложения.

## Ключевые таблицы `public`

| Таблица | Назначение |
| --- | --- |
| `tree_nodes` | Дерево проектов и документов |
| `node_files` | Файлы, привязанные к узлам дерева |
| `annotations` | Аннотации документов, связанные с `node_id` |
| `jobs` | OCR-задачи server-side режима |
| `job_files` | Файлы OCR-задачи: PDF, blocks, result и т.д. |
| `job_settings` | OCR-модели и настройки, зафиксированные на задаче |
| `stage_types` | Справочник стадий |
| `section_types` | Справочник разделов |

## Кто использует эти таблицы

- `app/tree_client/` работает с `tree_nodes`, `node_files`, `annotations`.
- `app/annotation_db.py` сохраняет и загружает аннотации через таблицу `annotations`.
- `services/remote_ocr/server/storage*.py` и `node_storage/` используют `jobs`, `job_files`, `job_settings`, `tree_nodes`, `node_files`.

## Практические правила

### Дерево проектов

- Узлы дерева хранятся в `tree_nodes`.
- Документные артефакты лежат в `node_files`.
- GUI получает их через `TreeClient`.

### Аннотации

- Аннотация — это сериализованный `Document`, сохраняемый в `annotations`.
- Канон формата и миграций лежит в `rd_core/annotation_io.py`.

### OCR server

- Статус и метаданные задач живут в `jobs`.
- R2-артефакты задачи отражаются в `job_files`.
- Выбранные модели фиксируются в `job_settings`.

## Как обновлять схему

Если меняется структура БД:

1. Обновите `database/migrations/prod.sql`.
2. Пересоберите экспорт в `database/exports/`.
3. Если изменилась прикладная модель, синхронизируйте код `tree_client`, `annotation_db` или server storage-слой.
4. Обновите этот файл только на уровне новых сущностей и связей, без полного SQL-справочника.

## Что не нужно дублировать в документации

В MVP-документации не нужно переносить сюда:

- полный SQL из `prod.sql`;
- все индексы и триггеры из экспортов;
- весь Supabase system schema dump.

Если нужен точный столбец, constraint или trigger, первичный источник — SQL и JSON-экспорты в `database/`.
